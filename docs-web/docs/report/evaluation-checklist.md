## 2. 리스크 서술 (3가지 이상 필수)

실제 프로젝트에서 겪은 리스크를 쓴다. 후보:

| 리스크 | 내용 | 대응 |
|---|---|---|
| 동시 예약 충돌 | 같은 서버에 동시 예약 요청 시 이중 점유 위험 | `version` 컬럼 낙관적 잠금 — 충돌 시 409, 정확히 1건만 성공 (테스트로 검증) |
| 프론트엔드 구현 지연 | 3인 팀에서 백엔드·테스트에 인력이 집중되어 프론트 구현이 늦어짐 | 화면 시안(frontend-design)과 IA 다이어그램을 먼저 확정해 API 계약을 선행 고정 |
| 실서버 부재 | 메트릭을 수집할 실제 GPU 서버가 없음 | server-pool 경량 에이전트로 N대 시뮬레이션 (/health·/metrics 계약 준수) |
| 비동기 스택 학습 곡선 | FastAPI 전면 async·asyncpg·testcontainers 등 처음 쓰는 기술 | 설계 문서를 먼저 쓰고(rule.md: 변경 시 문서부터) 작은 PR 단위로 리뷰 |

## 3. 느낀점

팀원 전원(팀장 포함 3명)이 각자 작성해야 한다. 분량보다 "설계 프로세스에서 무엇을 배웠는지"(요구사항 추적성, 동시성 제어, 문서-코드 동기화 등)가 드러나는 것이 좋다.

## 4. 상호평가 대비 메모

상호평가는 다른 조 학생들이 발표·보고서를 빠르게 훑고 채점한다. 즉 **시각 자료가 점수를 좌우**한다 — 실측 테스트 그래프, 추적성 매트릭스 한 장, 다이어그램, testkit 데모를 보고서와 발표 앞쪽에 배치할 것.
