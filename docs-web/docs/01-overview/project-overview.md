## 시스템 한 줄 정의

연구실·팀 단위 GPU/서버 공유 시 중복·유휴·블랙박스 문제를 해소하기 위한 **예약·할당·모니터링 플랫폼**.  
호텔 예약과 유사한 흐름으로, 학생/연구원은 서버를 예약하고 사용 후 반납하며, 팀 관리자는 Quota 한도를 관리하고, 서버 관리자는 인프라 전반을 운영합니다.

---

## 해결하는 문제

### 충돌 (Conflict)
- 여러 명이 동시에 같은 서버를 예약하려 할 때 **이중 점유 방지**  
- 낙관적 잠금으로 첫 요청만 성공, 나머지는 **대안 서버 자동 제시**

### 유휴 (Idleness)
- 예약했으나 사용하지 않는 자원 탐지  
- **1분 주기 모니터링**으로 CPU <5% 상태 30분 감지 → **경고 후 15분 회수**  
- 자동 회수로 팀의 Quota 회복

### 블랙박스 (Lack of Visibility)
- 누가 어떤 서버를 언제까지 쓰는지 불명확  
- **실시간 현황 대시보드** + **감사 로그 100% 기록**  
- **AIOps 이상 탐지·예측** — 장애 전에 알림

## 사용자 역할 (4종)

| 역할 | 약자 | 책임 | 주요 기능 |
|------|------|------|---------|
| **학생/연구원** | STU | 서버 예약·사용·반납 | F01~F08: 서버 현황·예약·대안·취소·반납·알림 |
| **팀 관리자** | MGR | 팀원 Quota 관리·초과 승인 | F09~F13, F21~F22: 승인함·Quota 설정·대시보드 |
| **서버 관리자** | ADM | 인프라 운영·보안·감사 | F14~F16, F20~F22: 서버 등록·삭제·점검·잠금해제·전체 관리 |
| **자동화 주체** | SYS | 스케줄러 잡 (별도 프로세스) | F23~F30: 메트릭 수집·유휴 회수·만료 반납·이상 탐지·건강 점수 |

## 핵심 특징

### 1. 예약 기반 자원 할당
- RESERVED → IN_USE → RETURNED/EXPIRED/RECLAIMED 상태 흐름  
- 서버의 `version` 컬럼으로 낙관적 잠금 → 동시성 문제 해결

### 2. AIOps 관점의 예측적 운영
- **이상 탐지** (μ±2σ) → **건강 점수** → **유휴 자동 회수**  
- (향후) 용량 예측·장애 열화 예측·이상 상관·LLM 설명

### 3. 실시간 알림 및 대시보드
- WebSocket으로 **예약 충돌 모달·알림 배지** 즉시 푸시  
- 운영 대시보드에 **스케줄러·자동 조치·헬스 상태** 5개 섹션 집계

### 4. 감사·보안
- **RBAC**(STU/MGR/ADM) 강제  
- **Rate limit** 초과 시 15분 잠금  
- **ADM/MGR 모든 조회·삭제·잠금 이벤트** 감사 로그
