1. 테스트 대상: 시스템 구조와 약점

구성요소내용
백엔드 (backend/) FastAPI(async) 단일 Uvicorn 워커 · PostgreSQL 16 + SQLAlchemy async · Redis 7(로그인 카운터, WS pub/sub) · JWT(HS256)+bcrypt · APScheduler 60초 주기 상태 전환
서버 풀 (server-pool/) 물리 서버를 흉내 내는 가상 에이전트 3대(포트 9101~9103) · GET /metrics는 동기 처리(스레드풀) · 자체 복구 로직 없음(장애는 docker stop/pause로 외부 주입) · 이미지에 hey, stress-ng 포함
주요 엔드포인트 POST /auth/login · POST /reservations · POST /reservations/instant · POST /reservations/{id}/cancel · POST /approval-requests/{id}/decision · GET /reservations · GET /teams/{id}/quotas · WS /ws/notifications
테스트에서 문제가 드러날 가능성이 높은 약점
  1. bcrypt가 동기(blocking) — 로그인/가입 해싱이 이벤트 루프를 막음. 동시 로그인 시 전체 지연.
  2. server.version 낙관적 락만 동시성 방어 — 같은 서버 동시 예약 시 한 건 빼고 409.
  3. quota.used는 락 없음 — 동시 요청이 쿼터 검사를 둘 다 통과하는 경쟁 상태. (가장 위험)
  4. DB 커넥션 풀 미설정 — 기본 ~10개에서 대기 병목.
  5. 스케줄러가 60초마다 전체 예약 스캔 — 데이터가 많으면 API와 자원 경합.
  6. WebSocket 연결마다 Redis 구독 1개 — 연결 수 증가 시 Redis 연결 고갈.

2. 성능 테스트 5종

차이는 “부하를 어떤 모양으로 주느냐”뿐이다. 공통 측정 지표: 응답시간 p50/p95/p99 · 처리량(RPS) · 오류율 · 자원 사용량(CPU/메모리/DB·Redis 연결 수).

테스트부하 형태1순위 대상핵심 확인
부하계단식 점진 증가 → 유지GET /reservations, /metrics정상 부하에서 응답 시간·처리량
과부하정점 2~3배 → 정상 복귀/reservations, /auth/login한계 동작 + 복귀 후 회복
스파이크 ★1~2초 내 급증 → 급강하/reservations/instant데이터 정합성(락·쿼터)
내구력저부하 장시간(6~24h) 유지전체 + 스케줄러/WS메모리·연결 누수
중단점 ★끝없이 계속 증가로그인·예약 경로성능 붕괴 임계점 수치

2.1 부하 테스트 load test

무엇을정상 사용 수준 부하에서 핵심 API의 응답 시간·처리량 측정
어떻게가상 사용자 10 → 50 → 100 → 200명 단계 증가, 각 단계 5분 유지
도구Locust — locust -f locustfile.py --users 200 --spawn-rate 20 --run-time 5m
합격 기준동시 50명에서 p95 < 300ms, 오류율 < 1%
확인 포인트DB 커넥션 풀(기본 ~10) 한계에서 응답 지연이 시작되는 지점

2.2 과부하 테스트 stress test

무엇을정점의 2배 이상 부하에서의 한계 동작과, 부하 해제 후 정상 복귀(graceful recovery)
어떻게정점 부하의 2~3배를 5~10분 → 정상 수준으로 복귀 → 응답시간·오류율 회복 관찰
도구Locust(부하 프로파일 교체) 또는 k6 stages — 300명 2분 → 50명 2분
합격 기준과부하 중 5xx가 나도 해제 후 1~2분 내 정상 응답률 복귀, 프로세스 다운 없음
확인 포인트bcrypt 동기 호출로 인한 이벤트 루프 포화 · 커넥션 풀 고갈 후 회복 여부

2.3 스파이크 테스트 spike test ★ 중점

무엇을순간 급증 부하(선착순 자원 쟁탈)에서 시스템 생존과 데이터 정합성
어떻게평상 부하(5명)에서 1~2초 안에 200명이 동시에 POST /reservations/instant 요청 → 직후 평상 부하 복귀 → DB에서 초과 배정·쿼터 초과 여부 검증
도구Locust --users 200 --spawn-rate 200 + 종료 후 DB 정합성 검증 스크립트
합격 기준서버 다운 없음 · 가용 서버 수보다 많은 예약이 성공하면 안 됨 · 쿼터 한도 초과 없음 · 직후 빠른 응답 복귀
확인 포인트quota.used 경쟁 상태(동시 요청이 한도를 넘겨 통과) — 여기서 드러날 가능성 최고. 낙관적 락이 초과 배정을 409로 막는지.

2.4 내구력 테스트 endurance test

무엇을장시간 운전 시 메모리 누수·자원 누적
어떻게동시 20명 수준을 6~24시간 유지하며 메모리·DB 커넥션·Redis 연결 수 추이 기록(30초 간격)
도구Locust(저부하 장시간) + docker stats 주기 수집
합격 기준메모리·커넥션 수가 우상향(누수)하지 않고 안정
확인 포인트WS 장시간 연결 시 Redis pub/sub 구독 누적 · 스케줄러의 커넥션 누적 점유

2.5 중단점 테스트 breakpoint test ★ 중점

무엇을동시 접속자를 계속 늘려 성능이 무너지기 시작하는 임계점 수치
어떻게0명부터 끊임없이 증가(예: 10분에 걸쳐 500명까지)시키며 p95·오류율 곡선 기록 → 무너지는 지점 산출
도구k6 ramping-vus · server-pool은 hey -z 30s -c <N> http://localhost:9101/metrics
합격 기준합격/불합격이 아니라 임계점 수치 자체가 산출물 — “동시 N명까지 p95<300ms 유지, 이후 급격히 악화”
확인 포인트로그인 경로는 bcrypt 때문에 낮은 동시성에서 임계점 예상 · 예약 경로는 409 비율 50% 초과 지점 · /metrics는 스레드풀(CPU수+4) 포화 지점

3. 보안 · 복원 · 회귀 테스트

3.1 보안 테스트 security test

무엇을인증·권한 체계가 부당한 접근을 막는지, 설정 약점이 없는지
어떻게
  • 인증 우회: 토큰 없음/만료/위조 → 401 (일부 테스트 존재)
  • RBAC 경계: STU가 /approval-requests 접근 → 403, MGR이 다른 팀 데이터 접근 차단 (미커버, 추가 필요)
  • 계정 잠금: 5회 실패 후 429 (테스트 존재)
  • 설정 점검: CORS *+credentials 오설정, 기본 JWT_SECRET 노출 여부
  • 입력 검증: 과거 시각 예약, 음수 쿼터 등 → 422
도구pytest + httpx (통합 테스트 스위트에 포함)
합격 기준권한 없는 접근 전부 401/403 · 5회 실패 시 429 · 기본 시크릿 미노출

3.2 복원 테스트 recovery test

무엇을일부러 장애를 일으킨 뒤 감지·복구·데이터 무손실 확인. server-pool이 최적 대상.
어떻게
  • docker stop agent-3 → 백엔드가 해당 서버를 MISSING으로 표시하는지
  • docker pause agent-3 → 타임아웃 경로에서도 MISSING 처리되는지
  • PostgreSQL·Redis를 내렸다 올림 → 백엔드 재연결 여부
  • 복구 후 예약 상태·쿼터 값 무손실 검증
도구docker stop/pause 기반 장애 주입 스크립트 + 백엔드 API 폴링 검증
합격 기준죽은 에이전트를 MISSING으로 정확히 표시 · 복구 후 데이터 무손실
확인 포인트백엔드에 명시적 타임아웃·재시도 없음 · 스케줄러가 예외를 조용히 삼켜 장애가 안 드러날 수 있음

3.3 회귀 테스트 regression test

무엇을코드 변경 후 기존 기능이 망가지지 않았는지
어떻게선택적 회귀: pytest -k reservation(변경 관련만) · 릴리스 전 retest-all: pytest 전체 · CI에서 매 PR 자동 실행
도구pytest + CI(GitHub Actions)
합격 기준변경 후 기존 테스트 스위트 전부 통과
확인 포인트Therac-25 사례처럼 “예전엔 문제없었다”고 회귀 테스트를 생략하면 안 됨

4. 정적 테스트 — 실행 없이 검토

4.1 검토 (peer review · walk-through · inspection)

무엇을실행으로 잡기 어려운 결함(특히 동시성)을 코드 리뷰로 조기 발견
어떻게PR 단위 2인 리뷰 + 체크리스트 기반 검사:
  • 데이터 결함 — quota.version 정의했으나 미사용
  • 제어 결함 — 예약 상태 전환 조건문 정확성
  • 인터페이스 결함 — /metrics 응답 스키마 ↔ 백엔드 파싱 일치
합격 기준체크리스트 검토 완료 · 발견 결함 문서화

4.2 자동화 정적 분석

무엇을코드·설정의 잠재 결함과 보안 취약점을 도구로 탐지
어떻게ruff check .(린트) · mypy app(타입) · bandit -r app(보안 패턴) · pip-audit(의존성 취약점) — CI에 통합
합격 기준린트·타입·보안 검사 통과, 알려진 취약 의존성 0건

5. 동적 테스트 — 실제 실행하며 검증

5.1 블랙박스 — 동등 분할 equivalence partitioning

무엇을입력을 유효/무효 클래스로 나눠 대표값으로 검증
어떻게쿼터: used < limit → 201, used ≥ limit → 422. 로그인 비밀번호: 정확/틀림/빈값. 
@pytest.mark.parametrize("used,code", [(LIMIT-1, 201), (LIMIT, 422)])
def test_quota(used, code): ...
도구pytest + httpx + testcontainers
합격 기준각 클래스 대표값에서 기대 상태코드 반환

5.2 블랙박스 — 경계값 분석 boundary value ★ 중점

무엇을오류가 몰리는 경계 지점 집중 검증 (off-by-one)
어떻게쿼터 limit-1 / limit / limit+1 · 로그인 실패 4회(통과)/5회(429 잠금) · 예약 시간 start==end, 과거 시각, 0분
도구pytest
합격 기준limit-1 성공/limit 실패 · 4회 통과/5회 잠금 · 비정상 시각 예약 거부

5.3 블랙박스 — 원인-결과 그래프 cause-effect

무엇을조건 조합표로 누락 케이스 도출
어떻게POST /approval-requests/{id}/decision — 원인: (PENDING?)×(역할 MGR/ADM?)×(같은 팀?)×(승인/거절). 결과: 승인→예약 생성+RESERVED / 권한 없음→403 / 비PENDING→409
도구pytest + 결정 테이블(decision table)
합격 기준조합표의 모든 유효 경로 + 예외 경로 검증

5.4 화이트박스 — 커버리지 기준

무엇을코드 내부 경로를 문장·분기·조건·기본경로 기준으로 검증
어떻게pytest --cov=app/services --cov-branch --cov-report=html — 낙관적 락 흐름(rowcount==0→409 vs 성공) 양쪽 경로, 상태 가드(if status != RESERVED) 양쪽 분기
도구pytest-cov
합격 기준핵심 서비스 모듈(reservation·auth·approval) 라인·분기 커버리지 80% 이상

6. V 모델 단계별 테스트와 커버리지 현황

단계우리 적용현황
단위 테스트서비스/유틸 함수 단위 (core/security.py 등)보안 함수 6건 존재 · 서비스 계층 보강 필요
통합 테스트 ★API↔DB↔Redis 연동. testcontainers로 실제 PG16+Redis7 기동/auth만 커버 · 예약/승인/쿼터 미커버 → 최우선 보강
시스템 테스트백엔드+server-pool+프론트 전체 연동. 성능 테스트(2절) 포함미실시
인수 테스트유스케이스(UC) 기준 사용자 시나리오 검증미실시 — 로컬 기동 후 시나리오 작성
영역자동화 테스트 현황
백엔드 — 인증 /auth커버됨 (~80%)
백엔드 — 예약(UC04~07) · 승인(UC09) · 쿼터(UC10) · 알림/WS · 스케줄러전부 미커버
서버 풀 — collector 값 범위/타입커버됨 (~70%)
서버 풀 — /metrics HTTP 레이어 · 동시 요청 · 컨테이너 통합미커버
