## 0. 목표 및 독자 우선순위

이 문서는 팀의 노션 워크스페이스 — 메인 페이지와 그 아래 6개 서브 페이지의 **정보 구조 · 레이아웃 · DB 스키마 · 운영 워크플로우**를 정의한다. 노션 페이지 자체를 만드는 작업은 본 설계에 따라 별도로 진행한다.

**독자 우선순위 (70 / 30)**

- **1순위 (70%): 팀 내부 (3명)** — 매일 들어와서 "오늘 뭐 하지" 답 + 회의록·결정사항 적기 + 명세서 작성
- **2순위 (30%): 교수 · 조교** — 평가·중간점검 시 "어떤 시스템이고, 산출물이 어디 있고, 어디까지 진척됐는지" 빠르게 파악

이 우선순위가 모든 레이아웃 결정의 기준이다. 의심스러우면 working hub 쪽으로.

## 1. 정보 구조 (사이트맵)

```
📄 메인 페이지 (서버 예약 / 할당 관리 시스템)
│
├── 📋 백로그             — Task DB · 4 views
├── 📐 시스템 설계        — 정적 페이지 · 다이어그램 3종 풀뷰

├── ⚙️  기능 명세         — Feature DB · UC 관계
├── 📅 회의록             — Meeting DB · 주 1회
└── 🎯 다음 회의 준비     — 정적 페이지 · 매주 갱신 → archive
```

## 2. 메인 페이지 구조

7개 섹션, **위→아래 순서 = 중요도 순서**.

### 2.1 Hero banner (최상단)

- 배경: 그라데이션 (남색 → 보라색), 흰 텍스트
- 상단 작은 라벨: `SOFTWARE ENGINEERING 01 · TEAM 4 · 2026 SPRING`
- 제목: `🖥️ 서버 예약 / 할당 관리 시스템`
- 1줄 소개: "연구실·팀 단위 GPU/서버 공유 시 중복·유휴를 막는 예약·할당·모니터링 플랫폼"
- Stats 배지 5개: `📅 Week N/14` · `👥 3명` · `🧩 20 UCs` · `⚙️ FastAPI + React` · `🗓 다음 회의: ${요일} ${시간}`
  - `Week N/14`, `다음 회의 일시`는 매주 회의 직후 수동 갱신

쇼케이스 30%의 가장 큰 슬롯. 교수·조교가 첫 1초에 "뭐 만드는 팀"인지 파악할 수 있게.

### 2.2 이번 주 포커스 + 빠른 액션 (2-column)

**왼쪽 (2/3 너비) — 이번 주 포커스 callout**
- 노란 배경 그라데이션
- 라벨: `🎯 THIS WEEK · W{n}` + 날짜 범위
- 3~5개 작업 항목 (`▶ 제목 — 담당자`)
- 매주 정기 회의 직후 수동 갱신

**오른쪽 (1/3 너비) — 빠른 액션 카드 3개**
- `📝 다음 회의 준비 페이지`
- `➕ 백로그에 태스크 추가`
- `🔍 명세서 검색` (노션 검색 단축)

### 2.3 전체 백로그 (Linked DB)

- 백로그 DB의 **Kanban 뷰** 임베드 (4컬럼: 대기 / 진행 / 리뷰 / 완료)
- "완료" 컬럼은 이번 주에 완료된 항목만 표시 (filter: 마감 ≥ 이번 주 시작일), 옛날 완료 항목은 noise
- 우측 상단 `전체 보기 →` 링크 (서브 페이지로)

### 2.4 시스템 설계 (다이어그램 썸네일 3개)

- 3컬럼 그리드, 각 카드: 썸네일 이미지 + 제목 + 메타 부제
- 카드별 부제:
  - 유스케이스 — `20 UCs · 4 actors`
  - 배치 — `Docker Compose · 단일 노드`
  - 컴포넌트 아키텍처 — `FastAPI · PostgreSQL · Redis`
- 카드 클릭 → `📐 시스템 설계` 서브 페이지

### 2.5 산출물 (6개 카드)

3컬럼 그리드, 각 카드에 **출처 라벨** 명시 (어디 가서 봐야 하는지 즉시 구분).

| 카드 | 출처 |
|------|------|
| 📄 프로젝트 계획서 | GitHub blob URL → `diagram-and-docs/docs/project_plan.md` |
| 🧱 기술 스택 명세 | GitHub blob URL → `diagram-and-docs/docs/tech_stacks.md` |
| 📘 유스케이스 명세서 | GitHub blob URL → `diagram-and-docs/docs/소프트웨어공학_01_4조_유스케이스_명세서.md` |
| 🔌 API 명세 DB | 노션 (UC 단위) |
| ⚙️ 기능 명세 DB | 노션 (UC 단위) |
| 📊 발표 자료 | 노션 (학기 말) |

### 2.6 회의록 · 결정사항 (Linked DB)

- 회의록 DB의 **Table 뷰** 임베드, 최근 5건 (sort: date desc, limit: 5)
- 컬럼: 날짜 · 제목 · 핵심 결정사항 · 참석자
- `주 1회 · 전체 보기 →` 링크

### 2.7 팀 + 외부 링크 (최하단, 회색 배경)

**왼쪽 (2/3 너비) — 팀**
- 3명 (👑 팀장 + 최민호 + 조동화)
- 챕 형태 (rounded badge)
- 팀장 이름은 본인 거 박을 것 (TODO)

**오른쪽 (1/3 너비) — 외부**
- GitHub (검정 배경 흰 글씨)
- Discord (보라 배경 흰 글씨)

## 3. 서브 페이지 설계

### 3.1 📋 백로그 (Task DB)

**Schema**

| 컬럼 | 타입 | 비고 |
|------|------|------|
| Title | text | 한 줄 요약 |
| UC | multi-select | UC01~UC21 (옵션 21개 미리 등록) |
| 담당 | person | |
| 상태 | select | 대기 / 진행 / 리뷰 / 완료 |
| 주차 | select | W1~W14 |
| 마감 | date | |
| 우선순위 | select | 높음 / 중간 / 낮음 |
| 레이어 | select | FE / BE / Infra / Docs |

**Views**

1. **Kanban** — 상태별 4컬럼 (메인 페이지에 임베드되는 뷰)
2. **Table (All)** — 전체 정렬·필터
3. **By UC** — UC별 grouped (명세서와 매칭 확인)
4. **This Week** — 주차 = 현재주 filter

### 3.2 📐 시스템 설계 (정적 페이지)

3 다이어그램 풀뷰. 각 다이어그램은 PNG export 임베드 + drawio 원본 GitHub 링크 + 짧은 설명 + 변경 이력.

1. **유스케이스 다이어그램** — 4 actors × 20 UCs
2. **배치 다이어그램** — Docker Compose 컨테이너 토폴로지 (API · Scheduler · PostgreSQL · Redis · Frontend)
3. **컴포넌트 아키텍처** — FastAPI 레이어 (api / core / infra / scheduler) 및 데이터 흐름

### 3.3 🔌 API 명세 (Endpoint DB)

한 row = 한 endpoint.

**Schema**

| 컬럼 | 타입 | 비고 |
|------|------|------|
| Endpoint | text | 예: `/api/reservations` |
| Method | select | GET / POST / PUT / DELETE |
| UC | relation | → 기능 명세 DB |
| Auth | select | STU / MGR / ADM / public |
| Request | rich text | code block (JSON) |
| Response | rich text | code block (JSON) |
| Status | multi-select | 200 / 201 / 400 / 403 / ... |
| 구현 상태 | select | 미정의 / 정의완료 / 구현완료 |

**Views**

1. **By UC** — UC별 그룹
2. **By Method** — GET/POST/...
3. **미구현** — 구현 상태 ≠ 완료

> 💡 FastAPI `/docs` Swagger UI와의 일치 여부를 주기적으로 회의 어젠다에 올린다. 어긋나면 명세 또는 코드 중 한 쪽을 갱신.

### 3.4 ⚙️ 기능 명세 (Feature DB)

한 row = 한 기능 (보통 UC 1:1).

**Schema**

| 컬럼 | 타입 | 비고 |
|------|------|------|
| UC ID | select | UC01~UC21 |
| 이름 | text | |
| 액터 | multi-select | STU / MGR / ADM / SYS |
| 설명 | rich text | |
| 입력/출력 | text | |
| 비즈니스 규칙 | bullet list | |
| Acceptance criteria | checklist | |
| 관련 API | relation | → API 명세 DB |
| 구현 상태 | select | 미정의 / 정의완료 / 구현완료 |

**Views**

1. **By 그룹** — 조회 / 알림 / 예약·할당 / Quota·승인 / 서버관리 / 자동화 / 보안·운영
2. **By Actor** — STU / MGR / ADM / SYS
3. **구현 진척도** — 구현 상태별 진행률 시각화

### 3.5 📅 회의록 (Meeting DB)

한 row = 한 회의 (주 1회).

**Schema**

| 컬럼 | 타입 | 비고 |
|------|------|------|
| 회차 | text | 수동 입력 (#01, #02, …) |
| 날짜 | date | |
| 주차 | select | W1~W14 |
| 참석자 | multi-person | |
| 제목 | text | |
| 핵심 결정사항 | rich text | 메인 페이지 임베드 시 표시 |
| 액션 아이템 | checklist | |

**Page template (각 회의록 페이지 내부 구조)**

1. 이전 회의 액션 아이템 검토 (relation)
2. 이번 주 진척 보고 (백로그 This Week 임베드)
3. 안건 (다음 회의 준비 페이지에서 가져옴)
4. 결정사항
5. 액션 아이템 (담당 · 마감 명시)
6. 다음 회의 일정

**액션 아이템 → 백로그 흐름 (수동)**

노션 native로 자동화 안 됨. 회의 끝나고 액션 아이템 중 task성 항목은 "백로그에 추가" 버튼(또는 수동 복사)으로 백로그 row 생성. 매뉴얼 단계로 합의함.

### 3.6 🎯 다음 회의 준비 (정적 페이지)

매주 갱신 → 회의 후 회의록 DB로 archive 흐름.

**구성**

1. 일시 (수정 가능, 기본은 다음 정기 회의)
2. 안건 (누구나 추가하는 checklist)

**Archive 흐름**

회의 시작 → 회의록 DB에 새 row 생성 → 이 페이지의 안건을 복사 → 회의 진행 → "다음 회의 준비" 페이지를 빈 템플릿으로 리셋.

## 4. 운영 워크플로우

### 4.1 주 1회 정기 회의 사이클

| 시점 | 행동 | 결과물 |
|------|------|--------|
| 회의 전 (수시) | "다음 회의 준비" 페이지에 안건 누적 | 안건 checklist |

| 회의 직후 | 메인 페이지 "이번 주 포커스" 갱신 | 메인 callout 갱신 |
| 회의 직후 | "다음 회의 준비" 페이지 리셋 | 빈 템플릿 |

## 5. 구현 메모 (노션 블록 매핑)

| 섹션 | 노션 블록 |
|------|----------|
| Hero banner | Callout (배경색 커스텀) + heading + paragraph + chip 배지 (인라인 코드 또는 colored mentions) |
| 이번 주 포커스 | Callout (노란색) + 토글 또는 체크리스트 |
| 빠른 액션 카드 | Column blocks + 페이지 링크 카드 |
| 백로그 임베드 | Linked database view (Kanban) |
| 시스템 설계 썸네일 | 3-column blocks + 이미지 + 페이지 링크 |
| 산출물 카드 | 3-column blocks + 카드형 페이지 링크 (북마크 또는 simple text + link) |
| 회의록 임베드 | Linked database view (Table, filter: top 5, sort: date desc) |
| 팀 · 외부 링크 | Column blocks + 텍스트 chip + 외부 URL 북마크 |

## 6. Out of Scope

본 설계는 노션 페이지 **구조**만을 정의한다. 다음 항목은 별도 작업.

- 다이어그램 실제 작성 (drawio 별도 작업)
- API · 기능 명세 실제 내용 채우기 (구현하면서 누적)
- 발표 자료 작성 (학기 말)
- 노션 외부 자동화 (Zapier, Discord 봇 연동 등 — 학기 범위 초과)
- 액션 아이템 → 백로그 자동 동기화 (노션 native 미지원, 수동으로 합의)
