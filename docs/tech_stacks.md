## 0. 의사결정 원칙

스택을 추가할지 결정할 때 본 팀이 따른 우선순위.

1. **명세 요구사항을 정확히 충족하는가** — 추측성 미래 요구로 스택을 늘리지 않는다.
2. **학기 단일 노드 데모 규모에서 운영 가능한가** — 운영급 분산 인프라(K8s, Kafka 등)는 학기 범위를 벗어남.
3. **팀이 학습/리뷰 가능한가** — 1~2명만 이해하는 스택은 도입하지 않는다.
4. **명시성 > 마법** — 트랜잭션·이벤트·스케줄러는 코드에서 흐름이 드러나는 쪽을 선호.
5. **추가했을 때 빠질 수 있는가** — 한 번 도입하면 운영 부담이 학기 내내 누적되므로 "지금 필요해진 시점에" 추가한다.

## 1. 채택된 스택

### 1.1 프론트엔드

#### React 18 + TypeScript 5
- **역할**: 학생/관리자/운영자 단일 SPA의 렌더링·상태 관리.
- **선정 이유**:
  - 컴포넌트 단위로 UC별 화면(예약 폼, 알림함, 운영 대시보드)을 분리하기 좋다.
  - TypeScript는 백엔드 OpenAPI 스키마를 `openapi-typescript`로 끌어와 요청/응답 타입을 컴파일 타임에 검증할 수 있어, FastAPI ↔ React 간 계약을 단단히 묶는다.
  - 팀원 다수가 가장 익숙한 프론트 스택.
- **대안 비교**: Vue/Svelte는 학습 자료는 충분하지만 shadcn/ui·Recharts·React Query 생태계의 풍부함을 포기해야 한다.

#### Vite 5
- **역할**: 개발 서버 + 번들러.
- **선정 이유**: HMR 속도가 매우 빨라 데모 중 코드 수정 → 즉시 반영 흐름이 자연스럽다. CRA는 사실상 deprecated.

#### shadcn/ui + TailwindCSS
- **역할**: 디자인 시스템 + 유틸리티 CSS.
- **선정 이유**:
  - shadcn은 라이브러리가 아니라 **컴포넌트 코드를 직접 복사해 쓰는 방식**이라 커스터마이즈 자유도가 높고 의존성 락인이 없다.
  - Tailwind는 UC별 화면이 많은 본 시스템에서 공통 spacing·typography를 강제하기 좋다.
- **대안 비교**: MUI/Ant Design은 빨리 만들 수 있지만 디자인이 획일적이고 데모용 커스터마이즈가 까다롭다.

#### Recharts
- **역할**: UC18(이상 징후 시각화)·UC19(헬스 점수)·UC21(운영 대시보드)의 차트 렌더링.
- **선정 이유**: React 친화적 API, 학습 곡선이 완만하고 학부 데모에 충분.
- **대안 비교**: D3는 표현력은 강하지만 학습 비용이 데모 가치를 초과. ECharts는 묵직하고 React 통합이 한 단계 더 필요.

### 1.2 백엔드 코어

#### FastAPI 0.115+ (Python 3.12)
- **역할**: REST API 서버, WebSocket 엔드포인트 호스트.
- **선정 이유**:
  - **async 네이티브**: DB·Redis·HTTP I/O가 많은 본 시스템 특성과 일치.
  - **Pydantic 통합**: 요청/응답 검증을 타입 선언만으로 강제 가능 → UC04~UC07 같은 입력 검증 케이스를 깔끔히 표현.
  - **OpenAPI 3.1 자동 생성**: 발표 데모에서 `/docs` Swagger UI를 그대로 시연 가능.
  - **WebSocket 기본 지원**: UC03-d 실시간 충돌 모달을 추가 라이브러리 없이 구현.
- **대안 비교**:
  - Spring Boot — 트랜잭션/낙관적 잠금 표현은 더 관용적이지만 JVM 기동 비용, Java 학습 비용이 큼.
  - Django — ORM과 admin은 강력하나 async가 부분적이고 OpenAPI 통합이 별도 패키지 필요.
  - Flask — 가볍지만 검증·문서화·DI가 모두 외부 패키지 조합이라 표준화가 어려움.

#### Uvicorn + gunicorn 워커
- **역할**: ASGI 서버.
- **선정 이유**: FastAPI 표준 조합. 단일 노드에서 워커 수만 조정해 멀티프로세스 확장.

#### Pydantic v2
- **역할**: 요청/응답 스키마, 설정 객체, DTO.
- **선정 이유**:
  - v2는 Rust 기반 코어로 검증 성능이 v1 대비 5~20배 향상.
  - `model_validator`, `Field` 제약으로 명세서의 "기간 ≤ 14일 / 사유 ≥ 20자" 같은 규칙을 선언적으로 표현.

#### SQLAlchemy 2.0 (`Mapped[]`) + Alembic
- **역할**: ORM + 마이그레이션.
- **선정 이유**:
  - **명시적 트랜잭션 경계**(`async with session.begin():`)가 코드에 그대로 드러나 SE 과목의 트랜잭션 학습 취지에 부합.
  - **낙관적 잠금**: `__mapper_args__ = {"version_id_col": "version"}`로 UC04~UC07의 동시성 제어를 표준 패턴으로 처리.
  - **2.0의 `Mapped[T]` 문법**: 타입 힌트 기반이라 IDE 자동완성·mypy 검증이 강력.
  - Alembic은 SQLA 팀이 직접 만든 마이그레이션 도구로, ORM 모델 변경 → 자동 후보 마이그레이션 생성 흐름이 자연스럽다.
- **대안 비교**: SQLModel은 SQLAlchemy 래퍼라 학습은 쉽지만 트랜잭션·관계 매핑이 복잡해지면 결국 raw SQLA로 내려가야 한다.

#### asyncpg
- **역할**: PostgreSQL async 드라이버.
- **선정 이유**: psycopg 대비 async 성능 우위. SQLAlchemy 2.0 async 엔진의 권장 드라이버.

### 1.3 인증

#### fastapi-users 또는 python-jose + passlib[bcrypt]
- **역할**: JWT 발급/검증, 비밀번호 해싱.
- **선정 이유**:
  - **fastapi-users**: 회원가입/로그인/리셋 라우터 일괄 제공 → 일정 단축.
  - **직접 구현 경로**: 학습 목적이 강하면 `python-jose`로 JWT·`passlib[bcrypt]`로 해싱을 직접 작성. 두 옵션 모두 의존성이 가볍다.
- **결정**: 초기 스프린트는 fastapi-users로 빠르게 가되, RBAC(STU/MGR/ADM) 분기는 직접 구현 권장.

### 1.4 데이터 계층

#### PostgreSQL 16
- **역할**: 단일 RDBMS. 사용자·예약·서버·승인·할당·알림·메트릭·감사로그 전부 수용.
- **선정 이유**:
  - **트랜잭션·MVCC 안정성**: UC04~UC07의 낙관적 잠금, 다중 행 일관성을 표준 SQL로 보장.
  - **파티셔닝**: `metrics_raw`를 `ts` 기준 월 단위 RANGE 파티셔닝으로 시계열 부하 흡수.
  - **JSONB**: 알림 payload, 감사 로그 등 가변 스키마 영역을 별도 저장소 없이 흡수.
  - **윈도우 함수 + 부분 인덱스**: UC18·UC19의 사전 집계 쿼리를 SQL 한 번으로 끝낼 수 있는 옵션 확보.
- **대안 비교**: MySQL은 윈도우 함수 도입이 비교적 최근이고 JSONB가 약함. TimescaleDB는 메트릭에 강하지만 학기 데모에 인프라 한 종 추가는 과함.

#### Redis 7 + `redis.asyncio`
- **역할**: 캐시, 분산 락, Pub/Sub, rate limit 카운터.
- **선정 이유**:
  - **캐시**: UC01·UC02의 서버 목록/예약 조회 hot path를 줄임 (`fastapi-cache2`로 데코레이터 캐싱).
  - **분산 락**: 스케줄러 잡(UC15, UC16)이 다중 인스턴스에서 동시에 돌지 않게 `SET key val NX PX ttl`로 단일 실행 보장.
  - **Pub/Sub**: UC03-d 실시간 알림을 다중 API 인스턴스 사이에 팬아웃.
  - **Rate limit (UC20)**: `INCR` + `PEXPIRE`로 슬라이딩 카운터를 원자적으로 구현.
- **대안 비교**: Memcached는 캐시만 가능, Pub/Sub·락·카운터를 별도 인프라로 분리해야 함. 한 종으로 다목적 처리되는 Redis가 운영 부담이 적다.

### 1.5 배치 / 스케줄러

#### APScheduler 3.x (단일 프로세스 데모) → Celery 5.x (멀티 워커 확장)
- **역할**: UC14~UC20의 주기 잡(메트릭 수집, 유휴 회수, 예약 만료, 승인 타임아웃, 이상 징후 탐지, 헬스 점수, rate limit 정리).
- **선정 이유**:
  - **APScheduler**: in-process 스케줄러로 학기 단일 노드 데모에 충분. cron 표현식과 interval 트리거 지원.
  - **별도 프로세스 분리**: API 서버와 스케줄러를 같은 코드베이스의 다른 컨테이너로 기동(`python -m app.scheduler.runner`) → API 요청 처리에 영향 없음.
  - **확장 경로 명시**: 잡이 무거워지거나 노드가 다중화되면 Celery beat(스케줄) + Celery worker(실행) + Redis broker로 교체. 잡 함수 시그니처는 호환되도록 설계.
- **대안 비교**:
  - **cron + 스크립트**: 가장 단순하지만 잡 상태/로그/재시도가 OS에 묶이고 코드와 분리됨.
  - **Celery 처음부터**: 브로커(Redis/RabbitMQ) + worker + beat 세 컴포넌트가 필요해 단일 노드 데모엔 과함.

### 1.6 실시간 푸시

#### FastAPI WebSocket + Redis Pub/Sub
- **역할**: UC03-d 충돌 모달 즉시 표시, UC03-a 결과 알림 푸시.
- **선정 이유**:
  - **WebSocket**: FastAPI 네이티브 지원이라 추가 라이브러리 없이 `@app.websocket("/ws/notifications")` 한 줄로 시작.
  - **Redis Pub/Sub**: API 인스턴스를 늘릴 때 한 인스턴스가 받은 알림을 다른 인스턴스의 WebSocket 연결로 팬아웃.
- **대안 비교**:
  - **SSE (Server-Sent Events)**: 단방향이면 충분. 양방향(서버 → 클라이언트만 필요한지) 요구가 확정되면 SSE로 단순화 가능. 현재는 향후 확장 여지를 위해 WebSocket.
  - **Long polling**: 구현은 단순하나 서버 자원/지연 모두 불리.

### 1.7 빌드 / 패키지

#### uv 0.5+
- **역할**: 의존성 잠금, 가상환경, 스크립트 실행.

#### Docker Compose
- **역할**: 로컬/CI에서 PostgreSQL·Redis·API·Scheduler·Frontend를 일괄 기동.
- **선정 이유**: 단일 노드 가정과 일치. K8s/Helm은 학기 범위를 벗어남.

### 1.8 테스트 / 품질

#### pytest + pytest-asyncio
- **역할**: 단위·통합 테스트 러너.
- **선정 이유**: Python 표준 사실상 단일 후보. async 테스트 지원.

#### Testcontainers-python
- **역할**: 테스트에서 실제 PostgreSQL·Redis 컨테이너를 띄움.
- **선정 이유**: 모킹된 DB는 실 마이그레이션/낙관적 잠금/쿼리 플랜을 검증하지 못함. UC04~UC07 같은 트랜잭션 경계 테스트에서 실 DB가 필수.

#### httpx
- **역할**: FastAPI 테스트 클라이언트, 외부 HTTP 호출.
- **선정 이유**: requests의 async 대응 + FastAPI `TestClient`의 백엔드.

#### ruff
- **역할**: 린터 + 포매터.
- **선정 이유**: flake8/black/isort/pyupgrade를 하나로 묶고 속도가 압도적으로 빠름. CI 시간 단축.

#### mypy (`--strict`)
- **역할**: 정적 타입 검사.
- **선정 이유**: Pydantic이 런타임 검증을 책임지지만, 코드 흐름의 타입 안정성은 mypy가 보완. 트레이드오프 표(2.2)에서 약속한 보완 수단.

### 1.9 CI/CD 파이프라인 (빌드·배포)

#### GitHub Actions
- **역할**: PR/머지 시 자동 테스트·린트·타입체크 + main 머지 시 빌드·배포까지 한 파이프라인으로 처리.
- **선정 이유**: 저장소가 GitHub이라 통합 비용 0. 학교 계정으로 무료 분 단위 충분.
- **파이프라인 구성**:
  - **CI 단계 (PR / push)**: `uv run pytest` (Testcontainers-python) + `ruff check` + `mypy app/` + 프론트 `tsc --noEmit` & `vitest`.
  - **빌드 단계**: 프론트엔드 `vite build` 결과물과 백엔드 Docker 이미지(`backend`, `scheduler`)를 워크플로우에서 동시 빌드해 GHCR(GitHub Container Registry)로 푸시.
  - **배포 단계**: main 머지 시 단일 노드 데모 서버에서 `docker compose pull && docker compose up -d`로 무중단 갱신.
- **대안 비교**: GitLab CI / Jenkins는 별도 호스팅이 필요해 학기 범위에서 운영 부담이 크다. 외부 CI(CircleCI, Travis)는 OSS 무료 한도 변동성이 있어 GitHub 내장 액션 쪽이 안정적.

## 2. 제외 / 도입 보류 스택

도입을 검토했지만 **명세 요구사항을 초과**하거나 **학기 범위에서 비용이 가치 초과**라고 판단한 스택. "지금 안 쓴다"는 결정이지 "영원히 안 쓴다"는 아니므로 재검토 트리거를 함께 기록한다.

### 2.1 RabbitMQ + aio-pika  ❌

- **검토 배경**: 초기에 UC03-a/UC03-d 알림 전송 채널 후보로 검토.
- **제외 이유**:
  1. **본 시스템의 "알림" 정의**: 명세서상 알림은 외부 채널(이메일/SMS/푸시)이 아니라 **앱 내부 `notifications` 테이블에 적재되는 인-앱 알림**이다. 메시지 큐로 라우팅할 외부 컨슈머가 존재하지 않는다.
  2. **트랜잭션 일관성**: 같은 트랜잭션에서 `session.add(Notification(...))` 한 줄이면 "예약 커밋되면 알림도 보이고, 롤백되면 같이 사라진다"가 자연히 성립. Outbox 패턴조차 불필요.
  3. **실시간 푸시 수단**: UC03-d는 메시지 큐가 아니라 **WebSocket**이 더 직접적인 도구. 큐를 거치면 오히려 지연·복잡도 증가.
  4. **운영 부담**: RabbitMQ 한 종 추가 = 컨테이너 + 관리 UI + 큐/익스체인지 토폴로지 설계 + 컨슈머 헬스 모니터링. 단일 노드 데모에서 회수되는 가치 없음.
- **재검토 트리거**:
  - 이메일/SMS/푸시 같은 **외부 채널** 알림 요구가 명세에 추가될 때.
  - 알림 처리에 **수초 이상의 무거운 작업**(예: PDF 첨부 생성)이 끼어들어 API 응답 경로에서 분리할 필요가 생길 때.
  - **다수의 외부 소비자**(분석 파이프라인, BI 도구 등)가 동일 이벤트를 구독해야 할 때.
- **대체 수단**: DB INSERT + WebSocket + Redis Pub/Sub. ([`project_plan.md` 5.3절](./project_plan.md#53-알림-uc03-a--uc03-d--db-기반-인-앱-알림))

### 2.2 Apache Kafka  ❌

- **제외 이유**: RabbitMQ와 동일한 사유 + 클러스터(ZooKeeper/KRaft) 운영 비용이 학기 범위를 크게 초과.
- **재검토 트리거**: 메트릭/이벤트 처리량이 초당 수만 건 규모가 되거나, 이벤트 소싱 아키텍처로 전환할 때. 현 명세는 분당 수회~수십회 수준이라 해당 없음.

### 2.3 TimescaleDB / InfluxDB  ❌

- **검토 배경**: UC14·UC18의 메트릭 시계열을 전용 저장소에 둘지 검토.
- **제외 이유**:
  1. PostgreSQL의 **파티셔닝 + BRIN 인덱스 + 사전 집계 테이블**(`metrics_5m`)로 학기 데모 데이터량(서버 ≤ 수십 대 × 분 단위)은 충분히 처리됨.
  2. 저장소 한 종 추가 = 백업/마이그레이션/접근 권한이 별도 라이프사이클로 복제됨.
  3. 단일 RDBMS 운영의 단순성이 학기 마지막 발표·인수인계에 유리.
- **재검토 트리거**: 메트릭 보존 기간이 1년 이상으로 늘거나, 다운샘플링 정책이 복잡해지는 시점.

### 2.4 NoSQL (MongoDB / DynamoDB)  ❌

- **제외 이유**:
  - 본 시스템의 핵심 데이터(예약, 승인, 할당)는 **강한 관계·트랜잭션·낙관적 잠금**이 필요한 영역이라 RDBMS가 자연스러움.
  - 알림 payload 같은 가변 스키마는 PostgreSQL JSONB로 흡수 가능.
- **재검토 트리거**: 스키마가 자주 바뀌는 비정형 콘텐츠(예: 사용자 자유 메모, 첨부 메타데이터)가 중심이 될 때.

### 2.5 GraphQL  ❌

- **제외 이유**:
  - 클라이언트가 단일(웹 SPA) → 멀티 클라이언트 호환을 위한 GraphQL 이점이 약함.
  - REST + OpenAPI 자동 문서화로 발표 데모 가치가 더 큼.
  - Pydantic ↔ GraphQL 스키마 이중 정의 부담.
- **재검토 트리거**: 모바일 앱이 추가되어 화면별 over-fetching이 문제가 될 때.

### 2.6 Spring Boot / JPA  ❌

- **검토 배경**: 표준 엔터프라이즈 스택으로 충분히 후보.
- **제외 이유**:
  - 팀 내 Python 경험자 비율이 더 높다.
  - 학기 단일 노드 데모 규모에서 JVM 기동 비용·메모리 부담이 무거움.
  - 트랜잭션 표현이 `@Transactional`로 짧아지는 장점은 있으나, SE 과목 학습 취지에는 **명시적 트랜잭션 경계**가 오히려 가시성이 좋다고 판단.
- **재검토 트리거**: 후속 학기에 멀티 서비스/멀티 팀으로 확장되고, JVM 생태계의 관측성 도구(Spring Boot Actuator 등)가 필요해질 때.

### 2.7 Kubernetes / Helm  ❌

- **제외 이유**: 단일 노드 가정과 부합하지 않음. Docker Compose로 충분.
- **재검토 트리거**: 무중단 배포·자동 스케일링이 요구사항으로 추가되는 시점.

### 2.8 Sentry / Datadog 같은 외부 SaaS 관측성  ❌ (학기 중)

- **제외 이유**: 비용·계정·데이터 외부 전송 이슈. 학기 중에는 컨테이너 로그 + Prometheus(선택) + PostgreSQL의 감사 로그 테이블로 충분.
- **재검토 트리거**: 운영 환경 이관 시.

### 2.9 ElasticSearch  ❌

- **제외 이유**: 본 시스템에 전문 검색(서버 설명 검색 등) 요구가 약함. PostgreSQL `tsvector`로 필요 시 대응 가능.
- **재검토 트리거**: UC21 운영 대시보드에서 로그/감사 검색 요구가 본격화될 때.

## 3. 향후 도입 결정 시 체크리스트

새로운 라이브러리/인프라를 추가하려는 PR이 올라오면 리뷰어는 다음을 확인한다.

- [ ] 어떤 UC/명세 요구사항을 충족하기 위함인가? (명세 줄 번호 인용)
- [ ] 같은 기능을 **이미 채택된 스택**으로 구현할 수 있는가? (Redis/PostgreSQL/FastAPI로 대체 가능 여부)
- [ ] 추가 시 **운영 부담**(컨테이너, 백업, 모니터링, 시크릿)이 무엇이며, 학기 범위에서 회수되는가?
- [ ] **제거 경로**가 있는가? (락인되는 의존성인지, 교체 가능한 어댑터로 격리 가능한지)
- [ ] 이 문서의 [2장 제외 목록](#2-제외--도입-보류-스택)에 이미 검토된 후보인가? 그렇다면 **재검토 트리거**가 충족되었는가?
